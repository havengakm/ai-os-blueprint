# Phase 2: aios-scout Extraction Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extract `systems/scout/` (57 Python files, 9 subdirs) from the monorepo into a new `aios-scout` pip package at `github.com/aios-kit/aios-scout` v0.1.0, rename namespace `systems.scout.*` → `aios.scout.*`, bump `aios-foundation` to v0.2.0 to add `BaseSystem`, and cut the monorepo over (~318 import-site rewrites + 50 test moves).

**Architecture:** Scout becomes pip-installable at `aios.scout.*` namespace, mirroring `aios.foundation.*`. The `BaseSystem`/`SystemResult` abstract base class moves to `aios-foundation` (v0.2.0 release) so scout can import it as a foundation primitive. Cross-system imports (beacon → scout backends) are bulk-rewritten via sed. Scout's own SQL migrations ship in the wheel for downstream provisioning.

**Tech Stack:** Python 3.11, uv, pytest, GitHub Actions, hatchling, sed for bulk rewrites, PEP 420 namespace packages.

**What is NOT in Phase 2:**
- `systems/beacon/`, `systems/optimizer/` (their carve-outs are Phase 2.2 and 2.3 — separate plans)
- Vault content migration (Phase 4)
- Deployment template repo (Phase 3)
- Cross-system orchestration changes (none needed; existing wiring works post-cutover)

**Phase 1 lessons applied:**
- Pip-installed package owns its namespace cleanly with `__init__.py` (regular package). Monorepo doesn't try to contribute to that namespace.
- Anything monorepo-only that referenced the namespace gets renamed/moved (Phase 1: `aios/foundation/registry.py` → `aios/dependency_container.py`). Phase 2: nothing to rename — `systems.scout.*` namespace fully moves to pip.
- `uv.lock` regenerated by reinstall during cutover; commit alongside.
- Hatchling needs `[tool.hatch.metadata] allow-direct-references = true` for `@ git+...` deps if monorepo had hatchling — currently does post-Phase-1.

```python
"""Smoke tests for BaseSystem + SystemResult.

Detailed system-behavior tests live with each system (e.g. aios-scout
tests ScoutSystem). These tests just confirm the abstract base contract
holds and the dataclass defaults behave."""

from aios.foundation import BaseSystem, SystemResult


def test_base_system_default_attributes():
    """Subclass without overrides preserves sensible defaults."""
    class StubSystem(BaseSystem):
        name = "stub"
        display_name = "Stub System"

    s = StubSystem()
    assert s.name == "stub"
    assert s.display_name == "Stub System"
    assert s.enabled is False  # default
    assert s.min_tier == "self_drive"
    assert s.foundation_context == {}


def test_base_system_handle_raises_unimplemented():
    """Default handle() raises NotImplementedError so subclasses must override."""
    import asyncio

    class StubSystem(BaseSystem):
        name = "stub"

    s = StubSystem()
    with __import__("pytest").raises(NotImplementedError):
        asyncio.run(s.handle("hi", "client", "user"))


def test_system_result_defaults():
    """SystemResult has empty defaults that don't share state across instances."""
    r1 = SystemResult()
    r2 = SystemResult()
    r1.follow_up_actions.append("a1")
    assert r2.follow_up_actions == []  # not shared


def test_tease_message_format():
    """tease_message uses display_name + description."""
    class StubSystem(BaseSystem):
        name = "stub"
        display_name = "Stub System"
        description = "does stub things"

    s = StubSystem()
    msg = s.tease_message()
    assert "Stub System" in msg
    assert "does stub things" in msg
```

# aios-scout Runbook

Operational guide for releasing, consuming, and rolling back `aios-scout`.

## Versioning

- Releases are git tags (e.g. `v0.1.0`). Tags are immutable.
- Breaking changes bump major. Additive changes bump minor. Patches bump patch.
- Foundation dep is pinned to a minor version range (e.g. `>=0.2.0,<0.3`); upgrade together when foundation cuts a major.

## Cutting a new release

1. Ensure all changes are committed and pushed to `main`.
2. Run the full test suite: `pytest -v`. Must pass.
3. Update version in `pyproject.toml`.
4. Commit the bump: `git commit -am "chore: bump version to vX.Y.Z"`.
5. Tag and push: `git tag vX.Y.Z && git push --tags`.
6. Create GitHub Release with changelog entry.
7. Notify downstream repos (deployment template + monorepo).

## Consuming the package

In any downstream `pyproject.toml`:

```toml
dependencies = [
    "aios-scout @ git+https://github.com/aios-kit/aios-scout.git@v0.1.0",
]
```

## Upgrade procedure for downstream

1. Read the release notes for the target tag.
2. Update the pin in your `pyproject.toml`.
3. Reinstall: `uv pip install --upgrade aios-scout` (or `uv sync`).
4. Run your full test suite. Failures here block the upgrade.
5. Run the daemon smoke test (`scripts/run_daemon_once.py --client-id=<client> --dry-run`). Verify decision_log writes still succeed.

## Rollback procedure

If a new scout release breaks production:

1. Pin `pyproject.toml` back to the previous tag.
2. `uv pip install --reinstall aios-scout`.
3. Run tests + daemon smoke to confirm green.
4. Open an issue against `aios-kit/aios-scout` with the failing test output for `vX.Y.Z+1`.

Tags are immutable. The prior version is always available. No data migration involved (scout is stateless except via Supabase, which is the deployment's concern).

## SQL migrations

`sql/migrations.sql` ships with the wheel. Apply via `psql` after install:

```bash
psql "$SUPABASE_DB_URL" -f $(python -c "import importlib.resources, aios.scout; print(importlib.resources.files('aios.scout').joinpath('../../sql/migrations.sql'))")
```

Or copy the file out of the installed package and apply directly. Migrations are idempotent (`IF NOT EXISTS`).

## Common issues

### `ModuleNotFoundError: No module named 'aios.scout'`

Reinstall: `uv pip install --reinstall aios-scout`. If still failing, verify the tag is pushed: `git ls-remote --tags https://github.com/aios-kit/aios-scout`.

### `ImportError: cannot import name 'X' from 'aios.foundation'`

Foundation pin is too low. Check `aios-foundation` version: `pip show aios-foundation`. Bump to the version `aios-scout` declares as a dep.

### Test failures referencing `systems.scout.*`

The downstream still has `from systems.scout.X import Y` somewhere. Update to `from aios.scout.X import Y`.

### Cloudflare scraper (Clutch source) fails in CI

CI doesn't have a display. Either skip live scraper tests in CI (mark with `@pytest.mark.skipif`) or run with `xvfb-run`. See `skills/playbooks/build-cloudflare-protected-scraper.md` in `aios-foundation`.

See [docs/public-api.md](docs/public-api.md).
