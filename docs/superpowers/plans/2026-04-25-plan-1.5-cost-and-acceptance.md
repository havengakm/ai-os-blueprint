# Plan 1.5: Cost Discipline + Acceptance Hardening + Body Template

**Branch**: `feat/plan-1.5-completion` (off main at `b1c060f`)
**Predecessor**: Plan 1 (foundation + Scout migration) merged at `1e77e1e`. Plan 1.5 Phase 0 (cost discipline + components v3 + niche scaffolding + memory closures) merged into main at `b1c060f` via two `--no-ff` merge commits (`4ed108d`, `b1c060f`) on 2026-04-25.
**Successor**: Plan 2 (Beacon send infrastructure). Strictly downstream; will not start until Task 1.5.12 acceptance passes.

## Context

Plan 1.5 was identified in `follow-ups-plan1.md` as the bridge between Plan 1 (Scout produces drafts) and Plan 2 (Beacon sends). It covers three things: (a) cost discipline so the enrich pipeline runs at MVP volume without blowing the budget; (b) rough edges raised by the Plan 1 acceptance run that need to land before Plan 2 ships; (c) the v2 storytelling body template that has been staged but lacked an architecture.

Phase 0 already shipped on `fix/cost-discipline-haiku-waterfall` (24 commits) and `chore/folder-cleanup-pre-plan15` (2 commits). Both merged to main on 2026-04-25. This plan documents Phase 0 for the historical record and specifies the remaining 12 tasks (Phase A through Phase E) on the new `feat/plan-1.5-completion` branch.

**Outcome of Plan 1.5**: AIOS produces validated, rendered outreach drafts at MVP cost for at least one ICP-aligned niche (`fitness_wellness`), with a body template architecture ready for bandit A/B between modular and storytelling formats. Plan 2 picks up at send infrastructure with no ambiguity about Plan 1.5 done-ness.

#### Task 1.5.6: Body template schema migration

**File**: `scripts/sql/013_body_template_component_type.sql` (new).

```sql
ALTER TABLE component_variants DROP CONSTRAINT IF EXISTS component_variants_component_type_check;
ALTER TABLE component_variants ADD CONSTRAINT component_variants_component_type_check
  CHECK (component_type IN (

    'credibility', 'offer_frame', 'cta', 'body_template'
```

#### Task 1.5.7: Composer body_template wiring

**File**: `systems/scout/outreach/composer.py`.

**Behavior**:
1. After bandit selects each inner component (icebreaker, bridge, pain_hook, who_i_am, credibility, cta), composer checks whether a `body_template` variant is enabled for the active niche/offer.
2. If yes: render each inner component's `variant_content` first, then substitute the rendered strings into the body_template's placeholders (`{{icebreaker_content}}`, `{{bridge_content}}`, `{{pain_hook_content}}`, `{{credibility_content}}`, `{{cta_content}}`).
3. If no: fall back to the legacy modular path (concatenate inner components in `_BODY_COMPONENTS` order).
4. Both paths route through `skills/meta/validate-writing.md` before returning.

**Acceptance**:
- [ ] Composer renders modular AND body_template paths cleanly.
- [ ] Decision log entries tag `body_template_id` when storytelling variant fires.
- [ ] Full regression on existing 532+ test suite.
- [ ] At least 8 new tests in `test_composer.py` for the body_template path.
