# Session: 2026-04-22

## Summary

Major architecture session. Designed and implemented the AIOS virtual-company structure: 15 skill categories, 7 departments, 3-tier skill model (capability / composite / playbook), global writing guardrails with validator, three-tier knowledge base (personal / company / experts), and a per-company silo rule. Refactored `context/` vs `data/knowledge/` to eliminate ambiguity. Adopted a project memory layer (this folder) based on Max Mitcham's system.

## Decisions Made

1. **Three-tier skill model**: capabilities (atomic) + composites (chains) + playbooks (E2E with human gates). Adapted from Gooseworks public catalog. All under `skills/`.
2. **Atomic skills rule**: one input → one output → one job. 15 capability categories, flat-by-category folders.
3. **Seven departments**: admin, finance, tax, legal, sales, marketing, operations. Manifests in `departments/`; skills stay in the shared `skills/` library.
4. **Global writing guardrails**: `rules/global-writing-guardrails.md` referenced by every content-producing skill; enforced by `skills/meta/validate-writing.md` on output.
5. **Per-company AIOS silo**: each company gets its own `context/` + `data/`; foundation (`skills/`, `rules/`, `departments/`, `agents/`, `systems/`, `aios/`, `scripts/`, `api/`, `config/`, `.claude/commands/`) is shared; content NEVER copies across deployments.
6. **Context vs knowledge tight rule**: `context/` = identity (brand, integrations, active projects); `data/knowledge/` = facts (personal, company, experts).
7. **Playbooks live under `skills/playbooks/`, not `data/reference/sops/`**: everything agent-invokable in one tree; `data/reference/sops/` stays for human-facing system docs.
8. **Memory layer adopted**: `memory/` folder with `MEMORY.md`, `INDEX.md`, `sessions/`. Sits alongside existing harness memory and `docs/superpowers/` without duplicating them.
