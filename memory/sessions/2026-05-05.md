## Update (Q2 answered + topology proposal on the table)

### Decisions Made
3. **Multi-repo, driven by ephemeral-routine cloning.** Operator confirmed: routines clone a repo to run, then the clone is deleted; trigger.dev jobs likewise need scoped clones. Each routine should pull only the slice (skills + context) it actually uses — Scout-sized for list-building, Content-sized for LinkedIn, etc. Not "premature decomposition" — it's required by the runtime model.
4. **Hard constraint = shared foundation.** Independent repos must still interact, learn, and optimise from each other via the AIOS foundation. "Lego blocks on a foundation." Decisions/findings flow into a shared data context for continuous evolution.

### Topology proposed (awaiting confirmation)
- `aios-foundation` (Python package) — autonomy, decision logger, memory, knowledge embedder, registry, feedback loop, Supabase client
- `aios-scout` / `aios-beacon` / `aios-optimizer` / `aios-content` — system repos, each imports `aios-foundation`, carries only its own skills
- `clymb-co-deployment` — per-deployment config + routine schedules, composes which systems run
- `clymb-co-brain` (Obsidian vault as git repo) — business knowledge
- `kirst-brain` (Obsidian vault as git repo) — personal + expert reference

Shared substrate = Supabase (decisions/memory/learnings tables). Foundation pkg = shared primitives. Together they give every system identical code AND identical data.
