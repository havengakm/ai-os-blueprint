## Plan 4 plan doc drafted: Karpathy/Saraev autoresearch framework

### Summary

Operator pasted Nick Saraev's 24-min YouTube walkthrough showing how he applied Karpathy's autoresearch (https://github.com/karpathy/autoresearch) to cold-email optimisation. Asked me to "create a similar test and structure for us." Drafted Plan 4 plan doc capturing the architecture (per-surface orchestrators, 3-file pattern, harvest→generate→deploy→measure→promote loop) adapted from both sources. Committed as `4353159` on `feat/plan-2-beacon`. Plan 4 implementation waits for Plan 2 Phase 2 + 3 + 5 + 30d reply data.

### Decisions Made

1. **Plan 4 plan doc lands NOW**, even though implementation waits for Plan 2 phases. Rationale: architecture is captured + team aligned + dependencies surfaced. Earlier reservation note ("plan doc drafted after Plan 2 Phase 5 ships") superseded by operator's explicit ask.
2. **5 per-surface orchestrators**, not 1 mega-orchestrator. Surfaces: subject_line, icebreaker, body_template, offer_frame, list_filter. Each runs independently with its own `baseline.md / resource.md / results.log` trio. Low coupling, easier debugging, parallel optimisation.
3. **3-file pattern preserved verbatim** from Karpathy + Saraev. The discipline (one file mutated, one curated, one append-only history) is the architectural insight worth keeping intact.
4. **Loop cadence**: 4-hour cron, 48-hour measurement window — matches Saraev's defaults (Karpathy's 5-min cycle is too fast for cold-email reply data).
5. **Significance threshold**: +0.5pp absolute OR +30% relative (whichever larger). Conservative — favours baseline when noise is high. Tunable per-client.
6. **GENERATE step is operator-interactive in v1** (Claude Code skill + Sonnet via Max credits per `feedback_max_credits_vs_api_boundary`). Promotes to daemon-autonomous in v2 after 30+ days of proven calibration. HARVEST + DEPLOY + PROMOTE are daemon-autonomous from day 1.
7. **8 phases over 2-3 weeks** of execution time once Plan 2 dependencies are met. Plan 4 won't start until Plan 2 Phase 5 ships.
8. **Plan 4 ≠ Plan 2 Phase 5 Optimizer**. Phase 5 = adjust priors on existing variants (operator approves). Plan 4 = autonomously generate NEW variant content + auto-promote winners. Both ship; they're complementary.
9. **Reuse Plan 2 infrastructure**: Beacon send adapter, outreach_send_log + outreach_reply, Optimizer cron + Slack scaffolding, decision_log emit pattern. No duplicated infra.
