### Decisions Made

1. **Slack delivery is best-effort, never blocks DB insert.** A Slack outage MUST NOT lose an escalation. EscalationRuntime sequence: DB insert → decision_log emit → Slack notify (try/except + log; failure swallowed). When `slack_notifier=None` (URL unset), the entire Slack path is a silent no-op.
2. **Inbox endpoints gated by `cron_secret_dep` for v1.** Operator uses the shared cron secret; per-user auth waits for the Next.js web app. Same pattern as `/api/pipeline/trigger`.
3. **Reused `decision_type='reply_handling'` for escalations + cool-off transitions** instead of adding new types. `reply_handling` exists from migration 001 — semantically fits both ("how we're handling this reply" / "how we're handling this contact's no-reply"). Avoided migration 020.
4. **Variant-pool-by-round deferred to Phase 5 follow-up.** Round-2-uses-different-copy is a composer/bandit-pull change; coupling Task 2.3.4 to the composer would have doubled scope. Acceptance box stays unchecked, plan doc notes the deferral.
5. **`run_cycle` runs re-entries before new cool-offs.** Frees the queue first (a contact who finished cool-off and re-enters won't be re-eligible for cool-off in the same cycle). Order is mostly cosmetic but matches operator intuition.
6. **Max rounds default = 4** per `feedback_surround_sound_architecture` ("3-4 rounds before dead"). Operator-tunable via constructor arg.
7. **CoolOffRuntime is purely orchestration; no LLM calls.** Cost is ~0c per cycle. Variant selection (which would change cost) is the Phase 5 piece.
